# GEO Retrieval Audit — 2026-04-27

First-time audit of how well KakeyaLattice is discoverable via
search engines (Google, Bing) and AI answer engines (Perplexity,
ChatGPT with search, Claude with search) on queries a real user
would issue when looking for an LLM KV-cache compression library.

Methodology: 10 web queries across three tiers, plus direct fetches
of the PyPI page and the GitHub repo page. All raw responses are
in the agent transcript for `docs: GEO audit + README star CTA +
repo-settings paste-ready text` — search that commit SHA to replay.

## Tier A — "knows the name"

The lower bound. If direct-name queries fail, the project is
invisible.

| query | first result | verdict |
|:------|:-------------|:--------|
| `kakeyalattice KV cache compression` | [libraries.io/pypi/kakeyalattice](https://libraries.io/pypi/kakeyalattice) | ✅ PASS — Libraries.io carries the full README, numeric tables intact |
| `"kakeyalattice" pypi` | libraries.io | ✅ PASS — first hit |
| `"KakeyaLattice" E8 lattice LLM` | libraries.io | ✅ PASS — synthesis correctly reports 2.77× on Qwen3-4B, 37.8 % advantage on GLM-4-9B-Chat |
| `KakeyaLattice transformers DynamicCache` | **MISATTRIBUTION** | ⚠️ FAIL — Perplexity conflated our project with an unrelated arXiv paper `arxiv.org/pdf/2504.05646` ("Lattice") on orthogonal state recurrence. The synthesis answered about that paper, not ours. |

**Tier A conclusion.** Input-side retrieval works for direct-name
queries. The misattribution is the first hard GEO risk: because we
have **no arXiv ID of our own**, AI engines pick a similarly-named
arXiv paper as the authoritative source and answer on its behalf.
Fixing this needs an arXiv ID — nothing else will.

bad: we *have* a vLLM plugin, but NexusQuant filed a public issue
(`vllm-project/vllm#39241`) and we did not, so retrievers only
find theirs.

## Tier C — "compare methods"

The question a procurement-stage reader asks.

| query | result |
|:------|:-------|
| `kakeyalattice vs nexusquant vs TurboQuant KV cache` | Perplexity synthesis names KakeyaLattice as "provides 2.4-3× better ratios than TurboQuant at matched perplexity" with correct per-model numbers — **but the links column goes to NexusQuant's PyPI / GitHub / DEV.to, not to us.** The answer cites us correctly but the referrable URLs are all NexusQuant. |

**Tier C conclusion.** We are recognised as a peer at the LLM
level, but the hyperlinks returned to a user point at competitors.
This is a symptom of the Tier A/B gaps: we do not yet sit on URLs
that AI engines treat as primary sources for the "lattice KV quant
methods" cluster.

## Competitive landscape as of 2026-04-27

NexusQuant (jagmarques) is the closest peer and is currently
winning the "lattice KV compression" query cluster. Their launch
pattern:

- ✅ `pip install nexusquant-kv` on PyPI
- ✅ 3 DEV.to posts with distinct long-tail keyword targets
- ✅ `vllm-project/vllm#39241` open issue with maintainer engagement
- ✅ HuggingFace Space demo
- ✅ 9 `awesome-*` list submissions, 4 merged
- ❌ **No arXiv preprint**  ← the one gap we can get ahead of

NVIDIA/kvpress is the category-leading implementation but is an
*eviction* library, not a quantisation library. It wins the generic
"best KV cache library" query because of the NVIDIA brand + 1 k
stars; it is not a direct competitor to KakeyaLattice on method
(we are compatible — eviction composes with lattice quant).
